# 04 — Layout engine — spec.md

> Read `00_constitution.md` and `shared/glossary.md` before this file.
> Depends on: `03_element_store`. Depended on by: `05_markup_style`, `07_components`.

## Purpose (one sentence)

Given a tree of layout nodes and the size of the window, compute the exact pixel rectangle
(x, y, width, height) of every node, using a flexbox + grid model ported from the Taffy
algorithm.

## Why it exists

Layout is the part of "the styling possibility of the web" that carries real expressive
power (per `INV-4.2`, we mimic Tailwind utilities, and those utilities are mostly a
front-end to flexbox/grid). This module is the algorithm those utility classes drive. It is
pure logic: no GPU, no signals, no markup parsing. That makes it the canonical test-first
module and the template every other spec copies.

## Inputs

1. A slice of `LayoutNode` (see `types.zig`), addressed by `ElementId` index — these live in
   the `ElementStore` arrays from module 03. This module READS `LayoutNode` data and the
   parent/child index structure; it WRITES only the `computed` rectangle field.
2. A root `Constraints` describing available space (normally the window's content size).

## Outputs

For every node reachable from the root, its `computed: Rect` field is filled with absolute
pixel coordinates relative to the root's origin (0,0 = top-left of the root). No other
state is mutated.

## Contract

The public API is defined exactly in `types.zig`. The single entry point is:

```zig
pub fn solve(store: *ElementStore, root: ElementId, available: Constraints) void;
```

`solve` performs the full layout pass and fills every reachable node's `computed` field.
It must be deterministic: same inputs → byte-identical outputs.

## Algorithm (normative summary; port of Taffy)

Two-phase, single pass down + single pass up per subtree (per `INV` performance intent —
every node visited O(1) times, no multi-pass reflow):

1. **Measure (down):** parent passes available `Constraints` to each child. Constraints are
   a `{min, max}` range on each axis.
2. **Resolve (up):** each child returns its chosen `Size` within those constraints; the
   parent places children according to its `direction`, `justify`, `align`, `gap`, and
   `padding`, then returns its own size.

Supported container models for v1:

- **Flex** (`display = .flex`): main axis = `direction` (`.row` | `.column`). Honors
  `justify_content`, `align_items`, `gap`, `padding`, and per-child `flex_grow`,
  `flex_shrink`, `flex_basis`.
- **Grid** (`display = .grid`): fixed track lists only — `grid_template_columns` and
  `grid_template_rows` as a list of `TrackSize` (`.px`, `.fr`, `.auto`). Honors `gap`,
  `padding`, and per-child `col_span` / `row_span`. Auto-placement fills row-major.
- **Block** (`display = .block`): single child stack, full available width, height = sum of
  children. The trivial fallback.

Sizing units supported: `.px` (absolute), `.percent` (of parent content box), `.auto`
(content-driven), `.fr` (grid fractional). See `types.zig` for the `Dimension` union.

## Non-goals (DO NOT implement — `INV-5.4`)

- NO `position: absolute/fixed/sticky`. Everything is in normal flow for v1.
- NO flex-wrap. Single-line flex only. (Wrapping is post-v1.)
- NO grid auto-tracks (`auto-fill`/`auto-fit`), NO named grid lines, NO grid areas. Fixed
  track lists only.
- NO min/max-content intrinsic sizing beyond what `.auto` requires for a leaf's measured
  size. Do not implement the full CSS intrinsic sizing spec.
- NO writing-mode, NO RTL (per `INV-1.3`). Main axis of a row is always left→right.
- NO baseline alignment. `align_items` supports `.start`, `.center`, `.end`, `.stretch`
  only.
- This module does NOT read or write any field other than `computed`. It does not touch
  styles, signals, or the dirty bitset.

## Edge cases (each has a test in `acceptance_test.zig`)

- **Empty container:** a flex/grid/block node with zero children gets its `computed` size
  from its own constraints (min if unconstrained), not a crash.
- **Zero available space:** `available = {0,0}` → every node computes a zero-area rect, no
  underflow, no negative dimensions ever.
- **Overflow:** children whose summed size exceeds the container do NOT shrink below their
  `min` and do NOT wrap; the container clips at paint time (not this module's concern) —
  here, children simply receive their min sizes and may extend past the parent's content
  box. Record their true rects; do not clamp them into the parent.
- **`flex_grow` distribution:** leftover main-axis space is distributed proportionally to
  `flex_grow` weights; if all weights are 0, leftover space is left unused (children stay at
  basis size), placed per `justify_content`.
- **Rounding:** all outputs are integer pixels. Round each node's position and size with
  round-half-to-even AFTER computing in floats, and ensure adjacent children do not overlap
  or gap by ±1px due to independent rounding (use cumulative-position rounding, not
  per-child).
- **Deep nesting:** must handle at least 64 levels without stack overflow; if recursion is
  used, document the depth bound, or use an explicit work stack.

## Performance intent (not a hard test, but a design constraint)

Linear in the number of nodes. No node is laid out more than a constant number of times.
No allocation during `solve` beyond a single reusable scratch buffer owned by the caller
(passed in or taken from the arena) — do NOT allocate per-node inside the pass.

## Reference

Port the algorithm from Taffy (Rust). Study its flexbox and CSS-grid track-sizing
implementations and reimplement the supported subset in Zig. Taffy is a *reference to
reimplement*, NOT a dependency to link (`INV-5.6`). Do not vendor Rust code; do not FFI
into it. Match its *results* for the supported subset where they overlap.
