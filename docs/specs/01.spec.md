# 01 — Platform spike — spec.md

> Read `00_constitution.md` before this file.
> Depends on: nothing but the Zig standard library, GLFW, and the Vulkan loader.
> Depended on by: the real render path (later). This is the FIRST module built.

## Purpose (one sentence)

Prove, on both Windows and Linux, that the chosen toolchain works end to end: open a GLFW
window, bring up Vulkan, present a cleared frame, and draw one triangle through a SPIR-V
pipeline — then tear everything down cleanly with zero validation errors.

## Why it exists

This is a de-risking spike (`INV` build order, module 01). Its only job is to surface the
hard parts of the stack in week one, before any engine is built on top of them. If Vulkan
setup or cross-OS surface creation is going to fight us, we find out now. The spike is
allowed to be ugly and hardcoded. Two artifacts survive into the real framework — the
`Platform` (window/surface/events) and the `VulkanBackend` skeleton (instance/device/
swapchain/present). The triangle and its pipeline are a throwaway proof, not a deliverable.

## Three notes that refine the constitution (record these)

1. **Scope refinement — the spike draws a triangle, not just a clear.**
   The constitution's build-order line says "clear color." A bare clear never exercises the
   shader/pipeline path, which is the riskiest part to defer. This spec REQUIRES one triangle
   through a SPIR-V graphics pipeline in addition to the clear, so the shader path is proven
   now. **Action: update the module-01 line in `00_constitution.md` to "clear color + one
   SPIR-V triangle."**

2. **Build-order consequence — module 01 cannot use module 03's `Size`.**
   The element store (module 03) is higher-numbered, so this module may not import from it.
   The spike defines its own pixel `Extent2D{ width, height: u32 }`. This is intentional and
   not a duplication problem: pixel extent (here) and layout `Size` (f32, module 03) are
   different concepts at different layers.

3. **Approved-tools addition — a GLSL→SPIR-V compiler at build time.**
   Compiling the triangle's shaders needs `glslc` (ships with the Vulkan SDK) invoked from
   `build.zig`, with the resulting `.spv` `@embedFile`'d. `INV-5.6` requires recording new
   build-time tools. **Action: add `glslc` (Vulkan SDK) to the approved build tools in
   `00_constitution.md`.** Do NOT add `shaderc`/`glslang` as a vendored library; use the SDK
   binary.

## The seam, done the cheap way (`INV-2.1` vs `INV-1.1`)

`INV-2.1` encourages leaving a seam for other GPU backends; `INV-1.1` forbids speculative
extension machinery. Resolve this by making the seam a **documented concrete API**, not a
vtable or runtime interface: `VulkanBackend` is a plain struct whose public method set is
the shape any future backend would match. Do NOT build a `GpuBackend` interface with
function pointers now. A future DX12 backend would be a sibling struct with the same method
names — that is the entire seam.

## Contract shape (note: different from the data modules)

For the data-oriented modules (03, 04) the struct field layout IS the contract. **Here it is
not** — `Platform` and `VulkanBackend` are thin wrappers over C libraries whose internal
fields are implementation-defined. Only the **public method signatures** in `types.zig` are
the contract. The agent may lay out internal fields however the implementation needs.

## What to build

`Platform` (GLFW-backed, `INV-2.2`):
- `init` with a window title and initial size; requests a Vulkan-capable window
  (`GLFW_CLIENT_API = GLFW_NO_API`).
- exposes the required Vulkan instance extensions (`glfwGetRequiredInstanceExtensions`).
- creates the `VkSurfaceKHR` for a given instance (`glfwCreateWindowSurface`).
- `pollEvents`, `shouldClose`, `framebufferSize` (`Extent2D`), clean `deinit`.

`VulkanBackend` (the only backend, `INV-2.1`):
- `init` from the platform's required extensions + surface: instance, debug messenger
  (validation layers on in debug builds), physical-device selection (a queue family with
  graphics + present, swapchain extension support), logical device + queues, swapchain +
  image views, render pass (or dynamic rendering), framebuffers, command pool + buffers,
  and the per-frame sync objects (image-available / render-finished semaphores, in-flight
  fence).
- `beginFrame` → acquire an image (handle `VK_ERROR_OUT_OF_DATE_KHR` by recreating the
  swapchain); `clear(Color)`; `drawTriangle()` (spike-only, hardcoded SPIR-V pipeline +
  vertex buffer); `endFrame` → submit + present.
- `onResize` / swapchain recreation.
- `deinit` that destroys everything in correct order with zero validation complaints.

## Non-goals (DO NOT implement — `INV-5.4`)

- NO `GpuBackend` interface / vtable / function-pointer indirection (see "the seam").
- NO DX12 or Metal backend (`INV-2.1`).
- NO draw-command-list abstraction yet (`INV-2.3` is real, but premature in a spike — the
  triangle is hardcoded; the command list arrives when components exist).
- NO text, no layout, no widgets, no theming. None of those modules exist yet.
- NO custom windowing/event layer — GLFW does it (`INV-2.2`).
- NO multi-window, no HDR, no MSAA, no depth buffer (2D UI, flat).

## Definition of done (this module is special — see smoke_test.zig + checklist.md)

Because the proof is partly visual, "done" has two halves:
- **Automatable:** `smoke_test.zig` runs N frames headlessly-as-possible on a machine with a
  GPU, asserting init succeeds, a device is selected, the swapchain has ≥1 image, frames
  submit/present with no error, validation layers report zero errors/warnings, and deinit
  leaks nothing. This is runnable, not pure-unit (it needs a real GPU/display).
- **Manual:** a human runs the spike on BOTH Windows and Linux and confirms a window appears
  showing the clear color with a visible triangle, resizes cleanly, and closes cleanly. The
  checklist records both confirmations. This is the one module where human-in-the-loop
  verification is part of done — do not pretend a `zig test` alone closes it.

## Reference

Follow the standard Vulkan bring-up sequence (e.g. vulkan-tutorial.com). It is a *reference
to follow*, not a dependency. Use validation layers from the start — they catch the vast
majority of bring-up mistakes and their silence is the strongest automatable signal of
correctness.
