# 08 — Schema forms — spec.md

> Read `00_constitution.md` before this file.
> Depends on: std (incl. std.json), modules 03/05/07. Depended on by: the app layer.
> This is the last v1 module and the product-value payload.

## Purpose (one sentence)

Turn a JSON Schema (known only at runtime) into a working form: walk it into a field model,
map each field to a widget, build the elements via module 07, bind each input to a path in a
dynamic `Value` tree, and validate the values against the schema.

## Why it exists

This is the runtime-dynamic path the whole architecture was built to make cheap (`INV-4.1`).
A BPM task, an admin-defined form — the renderer must handle a schema it has never seen.
Because the engine is data-oriented and the form logic is headless, the cost is small and the
core is fully testable without a GPU or font.

## The honest boundary (restating the "no miracle")

The comptime-baking advantage does NOT apply here — the schema arrives at runtime, so this
module ships real code: a schema model, a walker, a registry, a validator, and runtime path
binding. That is unavoidable. What makes it cheap is that it emits the same component-array
indices as the static path and runs as a linear pass.

## Architecture — headless core + thin scene glue (the pattern, one last time)

Four pure, testable pieces, then a thin binding layer:

1. **`Value`** — dynamic JSON-like tree (null/bool/int/float/string/array/object) with
   `getPath` / `setPath` over dotted paths (`"address.street"`, array index segments).
2. **Widget registry** — `widgetForNode(schema) WidgetKind`, reusing module 07's kinds.
3. **Walker** — `buildForm(schema)` → a flat `FormModel` (`[]FieldSpec`): one `FieldSpec` per
   leaf field, nesting expressed as a dotted `path`.
4. **Validator** — `validate(schema, value)` → `[]ValidationError` (`{path, message}`).

Then **`Form`** ties them to module 07: `mount(scene, tokens)` builds the elements and records
a `path → input ElementId` map; `getValue`/`setValue` read/write the `Value` tree;
`validate` runs the validator and exposes per-field errors. Everything except `mount` is
font-free and GPU-free, so the bulk of the tests are pure.

## v1 JSON-Schema keyword subset (this bounds the validator — the meatiest part)

**Supported in v1:**
- `type`: object, array, string, integer, number, boolean
- `format`: `date`, `date-time`, `email`, `uri` — validated by simple structural checks (no
  regex)
- `enum`: allowed-value list
- `required`: array of property names
- `properties` (object), `items` (array)
- `minLength` / `maxLength` (string), `minimum` / `maximum` (number/integer)
- `title` → field label; `x-widget` → widget hint override

**Deferred to post-v1 (explicit non-goals — these are the corner cases I flagged earlier):**
- `pattern` (arbitrary regex) — needs a regex engine, which is a dependency decision
  (`INV-5.6`); not in v1.
- `if` / `then` / `else` conditional schemas, `$ref` resolution, `dependentRequired`,
  `allOf`/`anyOf`/`oneOf` — the genuinely hard validator cases; deferred deliberately.

## Widget mapping (registry)

Using module 07's seven kinds (no checkbox/toggle in v1, so boolean uses a dropdown):
- `enum` present → `dropdown`
- `boolean` → `dropdown` (true/false)
- `string` (plain / email / uri / date) → `input`
- `integer` / `number` → `input`
- `object` → `column` (container; children are its properties)
- `array` → `column` (container; repeatable — v1 renders existing items, add/remove is a
  later concern)

The `x-widget` hint overrides the default (e.g. a long `enum` could hint a different widget),
which is what lets schema authors make forms feel hand-built rather than generated.

## Binding (`INV-4.1` — runtime path binding, the legitimate return of WPF-style resolution)

Static screens bind via comptime field offsets; here the data shape is unknown at compile
time, so binding is runtime path resolution into the `Value` tree. `Form.mount` records, per
input element, the dotted path it edits. `setValue(path, v)` / `getValue(path)` are the
read/write primitives. Path resolution is by string walk — correct and necessary here, not a
regression.

## Public API

```
Value (union) + getPath(path) ?*Value + setPath(alloc, path, v) !void
parseSchema(alloc, json: []const u8) !Schema          // via std.json
widgetForNode(schema) WidgetKind
buildForm(alloc, schema) ![]FieldSpec                  // the FormModel
validate(alloc, schema, value) ![]ValidationError

Form.init(alloc, schema) !Form        // builds model + empty values
Form.deinit
Form.mount(scene, tokens) !ElementId   // builds elements, records path->id (no font)
Form.getValue(path) ?*Value
Form.setValue(path, v) !void
Form.validate(alloc) ![]ValidationError
```

## Non-goals (DO NOT implement — `INV-5.4`)

- The deferred keywords above (pattern/if-then-else/$ref/combinators).
- NO array add/remove UI in v1 (render existing items only).
- NO new components — map only to module 07's seven kinds.
- NO submit/transport — `Form` edits a `Value`; persisting it is the app's job.
- NO comptime baking of schemas — schemas are runtime by nature (the honest boundary above).

## Edge cases (each has a test)

- `getPath`/`setPath` on nested objects; `setPath` creates intermediate objects; missing
  path → null.
- `widgetForNode`: enum→dropdown, boolean→dropdown, string→input, object→column.
- `buildForm` flattens nested objects into dotted-path `FieldSpec`s with labels from `title`.
- `validate`: required-missing, too-short string, out-of-range number, non-enum value, and a
  malformed email each produce one error; a fully valid value produces none.
- `Form.mount` builds one input per leaf field and records each path→ElementId; `set`/`get`
  round-trips through the `Value` tree.

## Performance intent

- `buildForm`, `validate`, and `mount` are linear in schema/field count.
- Allocations come from the form's arena; resetting a form resets the arena (per-screen
  lifecycle, `INV-3.5`).
