# 03 — Element store — spec.md

> Read `00_constitution.md` and `shared/glossary.md` before this file.
> Depends on: nothing but the Zig standard library. Depended on by: `04_layout_engine`,
> `05_markup_style`, `06_theme`, `07_components`, `08_schema_forms`.

## Purpose (one sentence)

Provide the single data-oriented store that holds every UI element's data in contiguous
parallel arrays, addresses elements by generational handle, models the parent/child tree as
index links, tracks which elements are dirty, and allocates everything in a per-screen
arena.

## Why it exists

This is the foundation the whole framework stands on (`INV-3.1`, `INV-3.2`, `INV-3.3`,
`INV-3.5`). Every other module reads and writes element data *through this store*. There is
no `Widget` object anywhere in the system — a widget is an index into these arrays. Get this
module right and the data-oriented architecture is real; get it wrong (e.g. by handing out
pointers or allocating per-widget) and every later module inherits the mistake.

## This module owns the shared element types

Because the store physically holds `LayoutNode` data and the constitution forbids module 03
from depending on any higher-numbered module, the canonical home for the element and
geometry types is **here**, in `03_element_store/types.zig`:

`ElementId`, `Rect`, `Size`, `Constraints`, `Insets`, `Dimension`, `TrackSize`, `Display`,
`FlexDirection`, `JustifyContent`, `AlignItems`, and `LayoutNode`.

Module 04 (layout engine) imports these from here; it does **not** redefine them. (If you
are reading an older `04_layout_engine/types.zig` that defines them locally, that is stale —
the reconciled version imports from module 03.)

## What the store holds (the data-oriented contract — `INV-3.1`)

Parallel arrays, all indexed by the same `u32` element index. Element `i` is the tuple of
`layout[i]`, `gen[i]`, `parent[i]`, and its links. There is **no** struct that bundles one
element's fields together on the heap.

- `layout: [N]LayoutNode` — the per-element layout/box data (the only "component" in v1;
  later modules add parallel arrays such as `style[]` and `text[]`, but module 03 ships
  only `layout`).
- `gen: [N]u32` — generation counter per index, for stale-handle detection.
- `parent: [N]u32` — parent index, or `NONE` for a root.
- `first_child`, `last_child`, `next_sibling: [N]u32` — the tree, as index links. Sentinel
  `NONE`. Children iterate in insertion order via `first_child` → `next_sibling` chain.
- `dirty` — a bitset over indices (`INV-3.3`).
- `free: [..]u32` — recycled indices available for reuse.
- `arena` — the per-screen `ArenaAllocator` backing the arrays (`INV-3.5`).

## Identity model (`INV-3.2`)

- A handle is `ElementId{ index: u32, gen: u32 }`.
- Allocating an element sets `gen[index]` and returns `{index, gen}`.
- `remove(id)` bumps `gen[index]` and pushes `index` onto `free`. Any previously-issued
  handle to that index now has a stale `gen` and `isValid` returns false.
- `get(id)` returns a `*LayoutNode` for **local use only** — never store it across frames
  (`INV-3.2`). It is the caller's responsibility not to hold the pointer past a mutation
  that could reallocate the arrays.

## Public API

Defined exactly in `types.zig`. Summary:

```
init(gpa)                  -> ElementStore        // production constructor
testInit(gpa)              -> !ElementStore        // test convenience (see note)
deinit(*self)              -> void
reset(*self)               -> void                 // per-screen: drop all, reset arena
addRoot(*self, LayoutNode) -> !ElementId
addChild(*self, parent, LayoutNode) -> !ElementId  // appends at tail of child chain
remove(*self, id)          -> void                 // recycle index, bump gen
isValid(*self, id)         -> bool
get(*self, id)             -> *LayoutNode           // local pointer; debug-asserts validity
childrenOf(*self, id)      -> ChildIterator         // insertion order
parentOf(*self, id)        -> ?ElementId
count(*self)               -> u32                   // number of live elements
markDirty(*self, id)       -> void
clearDirty(*self)          -> void                  // clears all dirty bits
dirtyIndices(*self)        -> DirtyIterator          // yields dirty element indices
```

`testInit` exists only so test files (including `04_layout_engine/acceptance_test.zig`) have
a one-call setup. It may simply call `init` plus a small preallocation. It is **not** a
production entry point — production code uses `init`.

## Behavior notes

- **Insertion order:** `childrenOf` MUST yield children in the order they were added via
  `addChild`. (Module 04's tests assert left-to-right placement that depends on this.)
- **Index reuse:** `addRoot`/`addChild` pop from `free` when available; a reused index keeps
  its bumped generation so stale handles stay invalid.
- **Dirty on add:** newly added elements are marked dirty (they have never been laid out).
- **reset vs deinit:** `reset` returns the store to empty for the next screen but keeps
  capacity (cheap arena reset); `deinit` frees everything including the arena.

## Non-goals (DO NOT implement — `INV-5.4`)

- NO multiple component arrays beyond `layout` in v1. (Style and text arrays arrive with
  modules 06/02; do not add them speculatively.)
- NO observer/callback/event-emitter change tracking. Dirty is a bitset, full stop
  (`INV-3.3`).
- NO pointer-based tree (no `*Element` children). Index links only (`INV-3.2`).
- NO thread safety / locking. Single-threaded UI for v1.
- NO serialization in this module (the flat layout makes it possible later; not now).

## Edge cases (each has a test in `acceptance_test.zig`)

- Removing an element and allocating a new one that reuses the freed index: the old handle
  must report `isValid == false`.
- `childrenOf` on a childless element yields nothing and does not crash.
- Growing past initial capacity (add hundreds of elements) must not corrupt earlier handles'
  validity (generations and indices stay consistent across array growth).
- `reset` then re-add: indices may be reused but the store behaves as freshly initialized.
- `dirtyIndices` after `clearDirty` yields nothing.

## Performance intent

- `addRoot`/`addChild`/`get`/`markDirty` are amortized O(1). `childrenOf` iteration is
  O(number of children). `remove` may be O(siblings) to unlink — acceptable; the dominant
  lifecycle is `reset`, not per-element removal (`INV-3.5`).
- No per-element heap allocation. Arrays grow geometrically from the arena/backing
  allocator; individual elements are never separately allocated or freed.
