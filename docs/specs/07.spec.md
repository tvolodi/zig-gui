# 07 — Components — spec.md

> Read `00_constitution.md` before this file.
> Depends on: std, module 02 (text), 03 (store), 05 (theme), 06 (markup+style). Depended on
> by: `08_schema_forms` and the app layer. This is the integrator.

## Purpose (one sentence)

Turn a parsed `NodeDesc` tree into a live element tree: map each tag to a widget kind,
resolve its classes (layered over per-kind defaults) into layout + style, store the result
in data-oriented arrays keyed by element index, and (in a separate pass) measure text.

## Why it exists

Everything below this module is a capability in isolation; module 07 is where they become
widgets. It is the first module that *instantiates* — it consumes module 06's descriptors and
module 05's tokens, writes `LayoutNode`s into module 03's store, holds per-element style, and
calls module 02 to measure text. The v1 component set is: Text, Button, Input, Card, Row,
Column, Dropdown.

## Build-order consequence — where per-element style lives (record this)

Module 03's spec anticipated "style/text arrays arrive with modules 06/02," implying they'd
be added to the `ElementStore`. That is **not possible**: `ComputedStyle` is defined in
module 05, and the store (03) may not import a higher-numbered module (`INV-3.4`). So the
store cannot hold a `[]ComputedStyle`.

**Resolved:** the presentation arrays (`kind[]`, `style[]`, `text[]`) live in module 07's
`Scene`, keyed by the same element index as the store's `layout[]`. A widget is still one
index across parallel arrays (`INV-3.1`) — the arrays are just split across two structs by
build-order necessity. `Scene` owns the `ElementStore` and is the single funnel for element
creation/removal, so the store's `layout[]` and the `Scene`'s arrays never desync.
**Action:** update module 03's note to point here instead of implying the arrays join the
store.

## Ownership rule (the sync contract)

- `Scene` owns the `ElementStore` (by value) and the parallel `kind`/`style`/`text` arrays.
- **All element creation and removal go through `Scene`**, never the store directly, so the
  arrays stay index-aligned (including on index reuse via the store's freelist).
- The layout engine (module 04) and renderer operate on the store via `Scene.store()` — they
  read/write `computed`, they do not create elements.

## Two passes (the headless split again)

1. **`instantiate` (no font, fully testable):** walk the descriptor tree; for each node, map
   tag → kind, build the per-kind default layout+style, layer the node's resolved classes on
   top, store kind/style/text, append the `LayoutNode` to the store, recurse children.
2. **`measurePass` (font-dependent, skip-able test):** for each text-bearing element, call
   module 02 to measure the text and fill `LayoutNode.measured`, ensuring glyphs are in the
   atlas. Kept separate so `instantiate` needs no font and unit-tests cleanly.

## Layering defaults under classes (the merge rule)

Per-kind defaults give a widget a sensible baseline (a `Button` looks like a button before
any class is added); classes override. Since module 06's `resolveClasses` returns full
structs (defaults for unset fields), use the **diff-against-empty** merge to avoid needing a
"which fields were set" bitmask:

```
base    = defaultStyleFor(kind, tokens) / defaultLayoutFor(kind)
resolved = resolveClasses(node.classes, tokens)
empty    = resolveClasses("", tokens)            // all class-defaults
final.field = if (resolved.field != empty.field) resolved.field else base.field
```

A class that sets a value equal to the empty-default won't override base — harmless, since it
equals the default anyway. Per-kind defaults come from module 05's component builders:
`button → buttonPrimary`, `card → cardSurface`, `input/dropdown → inputDefault`; `text`,
`row`, `column` have no style default. `row`/`column` set layout defaults (flex + direction).

## Public API

```
WidgetKind = { text, button, input, card, row, column, dropdown }
tagToKind(tag) ?WidgetKind                          // registry; unknown -> null

Scene.init(gpa) / deinit / reset
Scene.instantiate(desc, tokens) !ElementId          // builds tree (no font)
Scene.measurePass(font, atlas) !void                // fills LayoutNode.measured
Scene.store() *ElementStore                          // for layout/render
Scene.kindOf(id) WidgetKind
Scene.styleOf(id) *ComputedStyle
Scene.textOf(id) ?[]const u8
Scene.count() u32
```

## Non-goals (DO NOT implement — `INV-5.4`)

- NO components beyond the seven listed. (DataTable, charts, tabs, etc. are post-v1.)
- NO binding evaluation — `{bind ...}` attrs are recorded (text stored as the literal when
  present); resolving paths against state is module 08 / the state layer.
- NO event handling / interaction (click, focus, scroll) in v1 instantiation — that is a
  later concern; this module builds the static tree.
- NO adding arrays to the `ElementStore` (build-order — see above); `Scene` owns them.
- NO layout or painting here — module 04 lays out, the renderer paints.

## Edge cases (each has a test)

- Every known tag maps to its kind; an unknown tag → `tagToKind` returns null and
  `instantiate` returns an error (no silent skip).
- A `Button` with no classes gets the `buttonPrimary` default (background == accent); adding
  `bg-canvas` overrides the background to `bg_canvas` while keeping default padding.
- A nested tree instantiates with correct kinds, child order, and parent links in the store.
- `text="hi"` is stored as the element's text; an absent text attr → `textOf` returns null.
- After `reset`, the scene is empty and reusable; presentation arrays cleared in lockstep.

## Performance intent

- `instantiate` is O(nodes); allocations come from the scene's arena. Presentation arrays
  grow geometrically and are indexed directly (no per-element heap objects — `INV-3.1`).
- `measurePass` measures each unique glyph once (the atlas caches — module 02).
