## Right vs wrong (counter-examples — read before implementing)

**Identity / storage**
- RIGHT: read children via `store.childrenOf(id)` and address nodes by `ElementId`.
- WRONG: build a private tree of pointer-linked `LayoutNode` objects inside the module.
  Violates `INV-3.1` (data-oriented) and `INV-3.2` (handles, not pointers).

**What the module writes**
- RIGHT: write only `node.computed`.
- WRONG: cache a computed size back into `node.width` "to avoid recomputing." Mutating
  input fields corrupts the next frame and breaks determinism.

**Rounding**
- RIGHT: compute all positions in f32, then round cumulative edge positions so
  `child[i].x + child[i].w == child[i+1].x`.
- WRONG: round each child's width independently, then place them — produces ±1px gaps that
  show as seams at paint time (test 10 catches this).

**Overflow**
- RIGHT: an oversized non-shrinkable child gets its true rect even if it extends past the
  parent; clipping is the renderer's job, not layout's.
- WRONG: clamp the child's width to fit the parent. That silently corrupts intended layout
  and hides bugs (test 9 catches this).

**Scope**
- RIGHT: if a task asks for flex-wrap or absolute positioning, STOP and surface it — those
  are declared non-goals.
- WRONG: implement flex-wrap because it "seems natural for a layout engine." Scope creep
  across sessions is the primary failure mode (`INV-5.4`).
