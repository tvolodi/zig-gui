## Right vs wrong (counter-examples — read before implementing)

**Token discipline**
- RIGHT: `buttonPrimary(t)` sets `.background = t.accent`, `.radius = t.radius_md`,
  `.padding = .{ .left = t.sp_md, ... }`.
- WRONG: `.background = Color.hex(0x1D9E75)` or `.background = palette.accent_400` or
  `.radius = 8`. Hardcoding defeats re-theming and breaks dark mode (`INV-4.3`; tests 7–8
  catch it).

**Accent across modes**
- RIGHT: both `light` and `dark` set `accent = p.accent_400`; only `accent_hover` differs
  (darker stop in light, lighter in dark).
- WRONG: give light and dark different accents — the brand color drifts between modes
  (test 5 catches it).

**Layer crossing**
- RIGHT: tokens read palette; components read tokens; nothing reads two layers up.
- WRONG: a component builder that reads the palette directly, skipping tokens. The whole
  point of layer 2 is that layer 3 never sees raw values.

**Scope**
- RIGHT: ship the four component builders listed; add more only as real components need them.
- WRONG: build a generic theming engine, a `tokens.json` loader, or token animation now
  (`INV-5.4`).
