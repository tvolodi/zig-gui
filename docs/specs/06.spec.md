## Purpose (one sentence)

Provide the web-familiar authoring surface: a parser that turns `.ui` markup into a node
descriptor tree, and a Tailwind-subset resolver that turns a class string into a resolved
`ComputedStyle` + `LayoutNode` using theme tokens.

## Why it exists

This is the adoption strategy made real (the WPF lesson): developers write HTML-like markup
with Tailwind-ish classes, and it compiles to the native engine. Two pieces: the **parser**
(markup → descriptors) and the **resolver** (classes → resolved style/layout). The resolver
is the testable star; the parser is the fiddliest code in the project.

## Two mechanism refinements (record both)

1. **How baking happens — build-time codegen, not literal `comptime` parsing.**
   `INV-4.4` says `@embedFile`'d markup "becomes a typed struct tree at compile time." Its
   *intent* — no markup parser in the production binary, a baked struct tree — stands. But
   literal `comptime` parsing in Zig is impractical here because comptime has no general
   allocator, and the parser is allocator-based. **Resolved mechanism:** one allocator-based
   `parse` function, used two ways — (a) at **build time**, a small codegen step (our own
   `build.zig` tool) runs `parse` over the `.ui` files and emits a generated `.zig` file of
   `NodeDesc` struct literals (this is what ships — no parser in the production binary); (b)
   at **app runtime** behind `-Dhot-reload`, the same `parse` runs live for instant edits.
   **Action: update `INV-4.4` in `00_constitution.md`** to read "baked via a build-time
   codegen step that runs the same parser; a runtime parser exists only behind `-Dhot-reload`."

2. **The class scale split — colors/radius/type via tokens, spacing/sizing via fixed scale.**
   Tailwind's spacing is a fixed numeric scale (`p-4` = 4×4px); its color names are themeable.
   So the resolver maps **spacing, gap, and sizing utilities to a fixed px scale** (`n` → `n*4`
   px) and **color, radius, and font-size utilities to theme tokens**. This matches how
   Tailwind + design tokens actually compose and keeps theming on the themeable properties.
   Not a constitution change — just the documented rule for this module.

## Markup grammar (deliberately minimal — `INV-1.1`)

XML-like, attribute-based, no mixed-content text nodes:

```
node  := '<' TAG attr* ( '/>' | '>' node* '</' TAG '>' )
attr  := NAME '=' '"' value '"'
value := literal-text | '{bind' PATH '}'
```

- Tags are PascalCase component names (`Column`, `Text`, `Button`, …) — the parser does not
  validate which tags exist; that is module 07's job.
- `class="…"` is captured into `NodeDesc.classes`. All other attributes go into
  `NodeDesc.attrs`.
- An attribute value of the form `{bind some.path}` becomes an `AttrValue.bind`; anything
  else is `AttrValue.literal`.
- Text is an attribute (`text="hello"` / `text="{bind title}"`), NOT a text node between
  tags. Mixed content is out of scope for v1.

## Tailwind subset (v1 — the exact supported classes)

**Layout (write to `LayoutNode`):**
- `flex` → display=flex · `grid` → display=grid · `block` → display=block
- `flex-row` / `flex-col` → direction
- `justify-start|center|end|between` → justify_content (`between`→space_between)
- `items-start|center|end|stretch` → align_items
- `gap-{n}` → gap = n*4
- `grid-cols-{n}` → grid_template_columns = n equal `.fr` tracks
- `w-full` / `h-full` → width/height = percent 100 · `flex-1` → flex_grow=1, flex_basis=px 0

**Style (write to `ComputedStyle`):**
- `p-{n}` `px-{n}` `py-{n}` `pt-{n}` `pr-{n}` `pb-{n}` `pl-{n}` → padding = n*4 px
- `bg-canvas|surface|raised|accent|transparent` → background from tokens (or transparent)
- `text-body|muted|accent` → text_color from tokens
- `text-sm|base|lg` → font_size from tokens (`text_sm`/`text_base`/`text_lg`)
- `border` → border_width=1 · `border-subtle|default|strong` → border_color from tokens
- `rounded-sm|md|lg` → radius from tokens (`radius_sm/md/lg`) · `rounded-full` → radius=9999

**Rules:** classes are space-separated, order-independent EXCEPT last-wins on direct
conflict (`p-2 p-4` → 16). Unknown classes are ignored (no error, no crash). No cascade, no
specificity, no inheritance (`INV-4.2`).

## Public API

```
parse(allocator, source: []const u8) ParseError!NodeDesc      // root node
resolveClasses(classes: []const u8, tokens: Tokens) Resolved  // { style, layout }
```

`Resolved` is `{ style: ComputedStyle, layout: LayoutNode }` with only class-derived fields
set (rest at struct defaults).

## Non-goals (DO NOT implement — `INV-5.4`)

- NO literal `comptime` parser (use build-time codegen — refinement 1).
- NO mixed-content text nodes; text is an attribute.
- NO Tailwind features beyond the subset above (no responsive `md:` prefixes, no
  pseudo-states `hover:`, no arbitrary values `p-[7px]`, no color opacity `/50`).
- NO CSS cascade/specificity/inheritance/selectors (`INV-4.2`).
- NO tag validation or instantiation into the ElementStore — that is module 07.
- NO binding evaluation — the parser only RECORDS `{bind path}` as data; resolving paths is
  the state/forms layer's job.

## Edge cases (each has a test)

- Self-closing (`<Text/>`) and container (`<Column>…</Column>`) tags both parse.
- Nested children parse into the right tree shape and order.
- `{bind a.b.c}` becomes an `AttrValue.bind` with path `a.b.c`; a literal value stays literal.
- Mismatched/unclosed tags return a `ParseError`, not a crash.
- `resolveClasses` on `""` returns all defaults.
- Unknown class is ignored; surrounding known classes still apply.
- Conflicting classes: last wins (`p-2 p-4` → 16).
- Color/radius/font-size classes pull from tokens; spacing pulls from the fixed n*4 scale.

## Performance intent

- `resolveClasses` is O(classes), no allocation (writes into the returned structs).
- `parse` allocates the descriptor tree from the provided allocator/arena; at build time the
  codegen step owns that arena, at dev runtime the hot-reload arena does.
