## Right vs wrong (counter-examples — read before implementing)

**The boundary**
- RIGHT: defer `pattern`/`if-then-else`/`$ref`; ship the bounded v1 subset cleanly.
- WRONG: start implementing conditional schemas or pull in a regex library to support
  `pattern`. Both blow the v1 scope and the second adds an unrecorded dependency.

**Binding**
- RIGHT: `Form.mount` records each input's dotted path; `setValue`/`getValue` walk the
  `Value` tree at runtime.
- WRONG: try to resolve field paths at comptime — impossible here, the schema is runtime
  (`INV-4.1`; the honest boundary in spec.md).

**Reusing the engine**
- RIGHT: emit widgets through module 07's `Scene` (same component arrays as static screens).
- WRONG: build a separate parallel widget system for forms. The whole point is one engine,
  two front doors.

**Validator output**
- RIGHT: one `ValidationError{path, message}` per violation; empty slice = valid.
- WRONG: stop at the first error, or throw instead of collecting — forms need all errors at
  once to show every field's problem.

**Scope**
- RIGHT: render existing array items.
- WRONG: build array add/remove UI, submit/transport, or new widget types now (`INV-5.4`).
