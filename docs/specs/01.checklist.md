## Right vs wrong (counter-examples — read before implementing)

**The seam**
- RIGHT: `VulkanBackend` is a concrete struct; a future backend would be a sibling struct
  with the same method names.
- WRONG: define `GpuBackend` with function pointers and have `VulkanBackend` implement it
  "for flexibility." Speculative extension machinery violates `INV-1.1`.

**Validation layers**
- RIGHT: enable `VK_LAYER_KHRONOS_validation` + a debug messenger in debug builds; treat any
  message as a failure (`validationIssueCount` must stay 0).
- WRONG: develop with validation off and turn it on "later." Bring-up bugs then hide until
  much later and cost far more to find.

**Swapchain resize**
- RIGHT: handle `VK_ERROR_OUT_OF_DATE_KHR` from acquire/present by recreating the swapchain;
  `beginFrame` returns false so the caller skips the frame.
- WRONG: ignore the out-of-date result — the app corrupts or crashes on the first resize.

**Teardown order**
- RIGHT: destroy in reverse creation order, wait for device idle before destroying anything
  in use; deinit reports zero validation issues.
- WRONG: destroy the device before its swapchain/semaphores, or skip `vkDeviceWaitIdle`.
  Test 3 and the manual close-check catch this.

**Scope**
- RIGHT: hardcode the triangle; keep the spike ugly and minimal.
- WRONG: start building a real renderer, a draw-command list, or widget plumbing inside the
  spike. Those are later modules (`INV-5.4`).
