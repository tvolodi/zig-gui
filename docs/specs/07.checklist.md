## Right vs wrong (counter-examples — read before implementing)

**Where style lives**
- RIGHT: `Scene` holds `style: []ComputedStyle` keyed by element index, alongside the store's
  `layout[]`.
- WRONG: try to add `style` to the `ElementStore` in module 03 — that forces module 03 to
  import module 05, violating build order.

**Sync**
- RIGHT: `instantiate`/`reset` mutate the store and the parallel arrays together; index reuse
  writes at `id.index` in every array.
- WRONG: call `store.addChild` directly somewhere, bypassing `Scene` — the arrays desync and
  `styleOf`/`kindOf` return garbage for that index.

**Default/class layering**
- RIGHT: `final.field = if (resolved.field != empty.field) resolved else base` — classes
  override only what they actually set; defaults fill the rest.
- WRONG: replace the whole style with the resolved struct when any class is present (loses
  default padding/role), or ignore classes when a default exists (classes never apply).

**Scope**
- RIGHT: build the seven listed components as static elements.
- WRONG: add DataTable/charts/tabs, or wire click/focus/scroll handling now (`INV-5.4`).

**Binding**
- RIGHT: store literal `text="hi"`; leave `{bind ...}` for module 08 / the state layer.
- WRONG: evaluate a binding path against some state here.
