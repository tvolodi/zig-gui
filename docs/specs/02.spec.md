# 02 — Text — spec.md

> Read `00_constitution.md` before this file.
> Depends on: the Zig standard library and a TrueType rasterizer (stb_truetype — see note).
> Depended on by: `07_components` (which measures text and fills `LayoutNode.measured`),
> and the renderer (which uploads the glyph atlas and draws positioned glyphs).

## Purpose (one sentence)

Turn a UTF-8 string + a font + a pixel size into measured, line-wrapped, positioned glyphs
plus a packed glyph atlas the renderer can sample — for Latin and Cyrillic only.

## Why it exists

Text is the second-riskiest part of the stack after the platform spike, which is why it is
built early. The constitution deliberately shrank the risk: Latin + Cyrillic only, no
complex-script shaping, no bidirectional text, no HarfBuzz (`INV-1.3`). What remains is
tractable — glyph rasterization, a kerned advance model, and greedy word wrapping — and most
of it is pure CPU logic that unit-tests cleanly without a GPU or display.

## Two notes (one to record, one to accept)

1. **Approved-dependency addition — a TrueType rasterizer.**
   `INV-5.6` lists only GLFW, the Vulkan loader, and std. Rasterizing glyphs needs a font
   engine. Use **stb_truetype** (single-header, public domain) via `@cImport` — it does
   rasterization, metrics, and `kern`-table kerning, and is the minimal correct choice for a
   personal project (`INV-1.1`). Do NOT pull in FreeType (overkill) or attempt a from-scratch
   TTF rasterizer (needless risk). **Action: add stb_truetype to `INV-5.6` approved deps in
   `00_constitution.md`.**

2. **Capability caveat — `kern`-table kerning only.**
   stb_truetype reads the legacy `kern` table, not GPOS. Many modern fonts ship kerning only
   in GPOS and will report zero kerning. This is acceptable under "basic" text (`INV-1.3`) —
   document it, do not work around it. If a font's spacing looks loose, that is expected;
   pick a font with a `kern` table (e.g. DejaVu Sans, which also has full Cyrillic coverage).

## Build-order consequence

Module 02 is below the element store (03), so it may NOT import module 03's `Size`/`Rect`
(`INV-3.4`). It defines its own `TextExtent`. The handoff to layout is indirect: module 07
(components) calls `layoutParagraph`, reads the returned `TextExtent`, and copies it into the
element's `LayoutNode.measured` (a module-03 `Size`). Module 02 never touches the store.

## Contract shape

Two layers, deliberately separated (the headless-logic pattern):

- **Pure layout layer** — `measureWidth`, `wrap`, `blockHeight`. These operate on
  already-measured `Word` widths and font metrics, with NO font or GPU dependency. They are
  fully deterministic and are the core of `acceptance_test.zig`.
- **Font + atlas layer** — `Font` (stb_truetype-backed) and `GlyphAtlas` (CPU-side packed
  bitmap). `Font` rasterization needs a real font; `GlyphAtlas` packing is pure CPU and is
  unit-tested with synthetic glyph sizes. `layoutParagraph` ties everything together.

For `Font`/`GlyphAtlas`/`layoutParagraph` the struct field layout is implementation-defined
(they wrap a C library and a bitmap); only the public method signatures are the contract.
For the pure layer, the `Word`/`Line`/`TextExtent`/`FontMetrics` struct shapes ARE part of
the contract.

## What to build

Pure layer (no font, no GPU):
- `measureWidth(words, space_w)` — total advance of a run: sum of word widths plus one
  `space_w` between adjacent words.
- `wrap(words, space_w, max_width, out_lines)` — greedy word wrap at whitespace only.
  Returns the line count and fills `out_lines`. A single word wider than `max_width` gets its
  own line and is allowed to overflow (no mid-word breaking in v1).
- `blockHeight(line_count, line_height)` — total stacked height.

Font layer (stb_truetype):
- `Font.initFromBytes` / `deinit`; `metrics(px)` → ascent/descent/line_gap; `advance`,
  `kerning`; `rasterize(codepoint, px)` → grayscale bitmap + bearings + advance.

Atlas (pure CPU):
- `GlyphAtlas.init(w, h)` / `deinit`; `lookup(key)`; `insert(key, w, h, pixels)` packs a
  glyph and returns its `AtlasRect`, growing (doubling) the atlas when full rather than
  failing; `pixels()` exposes the bitmap for the renderer to upload.

Tie-together:
- `layoutParagraph(font, atlas, str, px, max_width)` → positioned glyphs (dest rect + atlas
  UV per glyph) and the paragraph `TextExtent`. Ensures each needed glyph is in the atlas.

## Non-goals (DO NOT implement — `INV-5.4`)

- NO complex-script shaping, ligatures, contextual forms, or GPOS (`INV-1.3`).
- NO bidirectional text, NO RTL.
- NO Unicode line-breaking algorithm (UAX #14) — whitespace breaking only.
- NO mid-word hyphenation or breaking.
- NO subpixel/LCD antialiasing, NO SDF glyphs (grayscale coverage only for v1).
- NO rich text (per-run styling, mixed fonts) — one font + one size per paragraph in v1.
- NO GPU upload here — the atlas is a CPU bitmap; the renderer uploads it.

## Edge cases (each has a test)

- Empty string / zero words → zero lines, zero extent, no crash.
- A single word wider than `max_width` → one line, overflows, does not loop forever.
- Exactly-fits boundary: words summing to exactly `max_width` stay on one line.
- Atlas insert that exceeds current size → atlas grows; existing entries remain valid
  (re-lookup returns correct rects).
- Duplicate `insert`/`lookup` of the same `GlyphKey` returns the same `AtlasRect` (caching).
- Inserted rects never overlap and always lie within atlas bounds.

## Performance intent

- Atlas insertion amortized O(1) with occasional O(n) repack on growth.
- `wrap` is O(number of words). No allocation inside `wrap` beyond the caller-provided
  `out_lines` buffer.
- Glyphs are rasterized once and cached in the atlas; never re-rasterized for the same key.
