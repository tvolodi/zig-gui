## Right vs wrong (counter-examples — read before implementing)

**One parser, two uses**
- RIGHT: a single allocator-based `parse`; the build step runs it to emit baked struct
  literals, and `-Dhot-reload` runs it live.
- WRONG: write a separate `comptime` parser and a separate runtime parser. Two parsers drift
  and double the bugs (spec refinement 1).

**Token vs fixed scale**
- RIGHT: `bg-accent` → `tokens.accent`; `rounded-md` → `tokens.radius_md`; `p-4` → `16` px.
- WRONG: `bg-accent` → a hardcoded hex (breaks theming), or `p-4` → some token step (Tailwind
  spacing is a fixed scale, not a token — tests check both).

**Unknown classes**
- RIGHT: silently ignore `totally-bogus`; apply the rest.
- WRONG: return an error or panic on an unrecognized class — that makes the authoring surface
  brittle (test catches it).

**Cascade**
- RIGHT: flat, order-independent set union with last-wins on direct conflict.
- WRONG: implement descendant selectors, specificity, or inheritance (`INV-4.2`).

**Binding evaluation**
- RIGHT: store `{bind user.name}` as `AttrValue.bind{ "user.name" }` and stop.
- WRONG: try to look up `user.name` against some state here — binding evaluation belongs to
  the state/forms layer, not the parser.
