# 09 — Renderer — spec.md

> Read `00_constitution.md` before this file.
> Depends on: std, module 01 (VulkanBackend/Platform), 02 (GlyphAtlas), 03 (Rect/ElementStore),
> 05 (ComputedStyle/Color), 07 (Scene). Depended on by: the app layer.
> This module completes the pipeline: scene + layout → pixels on screen.

## Purpose (one sentence)

Serialize a solved `Scene` into a flat `[]DrawCommand` list and submit it to the GPU via an
extended `VulkanBackend`, rendering colored quads, borders, and atlas-sampled glyphs.

## Why it exists

Modules 01–08 form a complete headless stack: markup is parsed, elements are instantiated,
layout is solved, styles are resolved. Nothing appears on screen because `VulkanBackend` can
only clear the framebuffer and draw a hardcoded triangle. Module 09 closes that gap — it
defines the draw-command vocabulary, implements the serializer that walks a solved `Scene`
into that vocabulary, extends the Vulkan backend with a pipeline capable of executing it, and
uploads the glyph atlas as a GPU texture. After module 09 the full path from `.ui` markup to
rendered pixels is complete.

## The boundary — INV-2.3 restated

The renderer must NOT know about widgets, layout nodes, or the element tree. It accepts a
`[]DrawCommand` and a `GlyphAtlas` (for the GPU texture) and nothing else. The serializer
(`buildDrawList`) knows about the scene; the renderer (`VulkanBackend`) does not. This
separation is load-bearing: it keeps the GPU code testable without a scene, and keeps the
serializer testable without a GPU.

## Draw command vocabulary

Three command kinds cover all v1 widget rendering:

```
DrawCommand = union(enum) {
    filled_rect: FilledRect,   // solid color quad, with optional corner radius
    border_rect: BorderRect,   // hollow quad (four edges), with optional corner radius
    glyph:       GlyphCmd,     // textured quad sampling one glyph cell from the atlas
}

FilledRect  = { rect: Rect, color: Color, radius: f32 }
BorderRect  = { rect: Rect, color: Color, width: f32, radius: f32 }
GlyphCmd    = { dst: Rect, uv: Rect, color: Color }
```

`Rect` is `{x, y, w, h}` in screen pixels, origin top-left. `uv` in `GlyphCmd` is in
normalized atlas coordinates (0..1). `Color` is `{r, g, b, a: u8}` (module 05's type).

## Serializer — buildDrawList

`buildDrawList(alloc, scene, atlas) ![]DrawCommand`

Walks the element tree in depth-first pre-order (painter's algorithm — parents behind
children). For each live element:

1. **Background:** if `style.background.a > 0`, emit `filled_rect` with `style.background`
   and `style.radius`.
2. **Border:** if `style.border_width > 0`, emit `border_rect` with `style.border_color`,
   `style.border_width`, and `style.radius`.
3. **Text:** if `scene.textOf(id)` is non-null AND `layout.computed.w > 0`, call
   `layoutParagraph` (module 02) to get positioned glyphs, then for each glyph emit one
   `glyph` command: `dst` is the glyph's pixel rect (origin + advance from layout), `uv` is
   the atlas cell in normalized coordinates, `color` is `style.text_color`.

The caller is responsible for `layout.solve()` before calling `buildDrawList`; the serializer
reads `layout.computed` but does not call solve itself.

**Allocation:** the returned slice is owned by `alloc`; the caller frees it. The total
command count is bounded by `3 * element_count + total_glyph_count` — linear, no per-element
heap objects (`INV-3.1`).

## GPU pipeline — what changes in VulkanBackend

Module 01's triangle pipeline is a spike and is explicitly scope-limited (see module 01
spec). Module 09 replaces it with a production quad pipeline by adding one new method to
`VulkanBackend`:

```
VulkanBackend.drawFrame(commands: []const DrawCommand, atlas: *const GpuAtlas) void
```

`drawFrame` is called between `beginFrame` / `endFrame` in place of `drawTriangle`. The
triangle pipeline and `drawTriangle` method remain in place and are not removed — they are the
module 01 smoke-test proof; removing them would break that test. `drawFrame` adds a second
pipeline alongside.

### New Vulkan objects required

| Object | Purpose |
|---|---|
| `GpuAtlas` | `VkImage` + `VkImageView` + `VkSampler` backed by the CPU atlas bitmap |
| Vertex/storage buffer | Per-frame quad geometry (one `QuadVertex[4]` per command) |
| Index buffer | Static 0,1,2,2,3,0 pattern per quad (or use `vkCmdDraw` with 6 vertices) |
| Descriptor set layout | Binding 0: combined image sampler (the atlas) |
| Push constants | `mat4 ortho` — orthographic projection from pixel coords to NDC |
| Quad pipeline | New `VkPipeline` with vertex input, alpha blending enabled, same render pass |

### Vertex layout

```
QuadVertex = { pos: [2]f32, uv: [2]f32, color: [4]u8, mode: u32 }
```

`mode`: 0 = filled_rect (uv ignored, output flat color), 1 = glyph (sample atlas, multiply by
color), 2 = border_rect (rendered as filled_rect in the fragment shader with border logic
handled by emitting four thin quads from the serializer — see below).

**Border implementation:** the serializer expands each `border_rect` into four `filled_rect`
quads (top/right/bottom/left edges) before emitting GPU vertices. This avoids a more complex
SDF border shader and keeps the fragment shader simple. The `border_rect` command kind exists
in the CPU vocabulary for clarity; by the time data reaches the GPU it is already four quads.

### Shaders

Two GLSL shaders, compiled to SPIR-V by `glslc` at build time (same mechanism as module 01):

**Vertex shader** (`quad.vert`): reads `QuadVertex`, multiplies `pos` by the ortho push
constant, passes `uv`, `color`, and `mode` to the fragment shader.

**Fragment shader** (`quad.frag`): if `mode == 1`, samples the atlas at `uv` and uses the
red channel as an alpha mask (stb_truetype atlas is single-channel); otherwise outputs the
flat `color`. Alpha blending must be enabled in the pipeline state
(`srcAlphaBlendFactor = SRC_ALPHA`, `dstAlphaBlendFactor = ONE_MINUS_SRC_ALPHA`).

### GpuAtlas upload

`GpuAtlas.upload(gpa, device, phys_device, cmd_pool, queue, atlas: *const GlyphAtlas) !GpuAtlas`

Uploads the CPU atlas bitmap to a `VK_FORMAT_R8_UNORM` `VkImage` via a staging buffer, issues
a layout transition (`UNDEFINED → SHADER_READ_ONLY_OPTIMAL`), and creates the `VkImageView`
and `VkSampler`. The staging buffer is freed after the upload command completes.

`GpuAtlas.deinit(device)` destroys image, view, and sampler.

The atlas may grow during a `measurePass` call (new glyphs rasterized). Re-upload is triggered
by comparing `atlas.generation` before and after `measurePass`; if changed, call
`GpuAtlas.upload` again. The generation counter is a `u32` on `GlyphAtlas` incremented each
time the atlas bitmap is modified.

**Action:** add `generation: u32` to `GlyphAtlas` in module 02's `types.zig` (module 09
specifies this addition; module 02 implementer applies it, or module 09 implementer adds it
as a patch). This is a backwards-compatible field addition — no existing signature changes.

### Per-frame buffer strategy (v1)

Upload all quad vertices for the current frame into a single host-visible, host-coherent
`VkBuffer` (no staging for per-frame geometry — simplest correct approach for v1). Map the
buffer, write the vertices, unmap, draw. The buffer is pre-allocated for `MAX_QUADS = 65536`
quads; if a frame exceeds this, the serializer truncates and logs a warning (post-v1:
dynamic growth). This avoids GPU synchronization complexity at the cost of a fixed ceiling.

## Public API

```
// CPU types
DrawCommand (union: filled_rect / border_rect / glyph)
FilledRect, BorderRect, GlyphCmd

// Serializer (pure CPU, no GPU, fully testable without Vulkan)
buildDrawList(alloc, scene: *Scene, atlas: *GlyphAtlas) ![]DrawCommand

// GPU atlas wrapper
GpuAtlas.upload(gpa, device, phys_device, cmd_pool, queue, atlas) !GpuAtlas
GpuAtlas.deinit(device) void

// Backend extension (added to VulkanBackend in module 01's types.zig)
VulkanBackend.drawFrame(commands: []const DrawCommand, atlas: *const GpuAtlas) void
VulkanBackend.initQuadPipeline(gpa) !void   // called once after VulkanBackend.init
VulkanBackend.deinitQuadPipeline() void     // called before VulkanBackend.deinit
```

`drawTriangle` and the triangle pipeline from module 01 remain untouched.

`build.zig` gains:
- `glslc` invocations for `quad.vert` and `quad.frag` (same pattern as the existing triangle
  shaders), bundled into the same `WriteFiles` step or a new one.
- Module 09 declared as `b.addModule("renderer", ...)` importing modules 01/02/03/05/07.
- `zig build test-09` step running `docs/specs/09.acceptance_test.zig`.
- `zig build test-09-unit` step running `src/09/09_test.zig`.

## Non-goals (DO NOT implement — INV-5.4)

- NO clipping / scissor regions per element in v1 — one full-viewport scissor for the whole
  frame is sufficient.
- NO depth buffer / z-ordering beyond painter's order (pre-order DFS gives correct parent-
  behind-child ordering; overlapping siblings at the same depth are out of scope).
- NO shadow, gradient, or image-fill commands in v1 — flat color and atlas glyphs only.
- NO font atlas streaming (multiple atlas pages) — one `GlyphAtlas` of fixed size per session.
- NO dynamic vertex buffer growth — `MAX_QUADS = 65536` is a hard ceiling with a warning.
- NO separate render thread — `drawFrame` is called synchronously on the main thread.
- NO interaction / hit-testing — this module paints; input handling is post-v1.

## Edge cases (each has a test)

- An element with `background.a == 0` and `border_width == 0` and no text emits zero
  commands (invisible containers, e.g. a `Row` with default style).
- An element with `computed.w == 0` or `computed.h == 0` (not yet laid out, or collapsed)
  emits zero commands — do not emit zero-area rects.
- Text that exceeds `computed.w` is clipped by emitting only glyphs whose `dst.x + dst.w <=
  rect.x + rect.w` (no partial-glyph scissor; just skip overflow glyphs).
- A `border_rect` with `width` larger than half the rect's minimum dimension is clamped to
  `min(w, h) / 2` before expanding to four quads (prevents inverted geometry).
- `GpuAtlas.upload` called twice (atlas regenerated after measure) — old resources are
  destroyed before new ones are created; no resource leak.
- Frame with zero draw commands — `drawFrame` is a no-op (no draw call issued); valid.

## Acceptance test structure

The acceptance test is split into two sections matching the headless-split pattern:

**Pure CPU tests (no GPU required):**
- `buildDrawList` on a known solved scene produces the correct command sequence (kinds,
  colors, rects match the expected values from a manually solved layout).
- Invisible element (alpha 0 background, no border, no text) → zero commands.
- Zero-size element → zero commands.
- Border expansion: one `border_rect` command in the draw list expands correctly to four
  filled rects for GPU submission (tested via the quad-expansion helper, not via Vulkan).

**GPU tests (skip if no Vulkan display):**
- `VulkanBackend.initQuadPipeline` succeeds on a real device.
- `drawFrame` with a minimal command list completes without Vulkan validation errors
  (`backend.validationIssueCount() == 0`).
- `GpuAtlas.upload` + `drawFrame` with a glyph command completes without validation errors.

## Performance intent

- `buildDrawList` is O(elements + total glyphs): one pass, no allocations beyond the output
  slice, no per-element heap objects (`INV-3.1`).
- Per-frame vertex upload is a single `memcpy` into the mapped buffer — no per-quad GPU calls.
- Atlas upload (re-upload on generation change) is amortized: typically once at startup and
  once after the first `measurePass`; zero cost on frames where the atlas does not change.
