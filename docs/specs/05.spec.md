# 05 ‚Äî Theme ‚Äî spec.md

> Read `00_constitution.md` before this file.
> Depends on: the Zig standard library and module 03 (for `Insets`). Depended on by:
> `06_markup_style` (the Tailwind resolver reads tokens) and `07_components`.

## Purpose (one sentence)

Implement the four-layer token model ‚Äî raw palette ‚Üí semantic tokens ‚Üí component styles ‚Üí
resolved `ComputedStyle` ‚Äî so that swapping a theme changes only the palette/mode and every
widget re-themes coherently, with light and dark both supported.

## Build-order correction (record this ‚Äî it reorders the constitution)

The constitution's build-order list currently has **05 = markup + style** and
**06 = theme**. That is backwards: the Tailwind-subset resolver in the style module maps a
class like `bg-accent` to a concrete color by reading a **token**, so the token model must
exist first. A lower-numbered module may not depend on a higher one (`INV-3.4`), so theme
must precede markup+style.

**Action:** in `00_constitution.md`, swap those two build-order lines so they read:
```
05  theme            ‚Äî four-layer token model, light/dark
06  markup + style   ‚Äî comptime .ui parser, Tailwind-subset resolver
```
This spec is numbered 05 to reflect the corrected order. The markup+style spec, when
written, becomes 06.

## The four layers (`INV-4.3`)

1. **Palette** ‚Äî raw named values (gray scale, accent ramp, status colors, base spacing,
   type scale). The ONLY layer that changes between brand themes. Nothing outside layer 2
   reads it.
2. **Tokens** ‚Äî semantic *roles* (`bg_canvas`, `text_body`, `accent`, `border_default`,
   spacing scale, radii, type sizes). Built from a palette + mode via `Tokens.light(p)` /
   `Tokens.dark(p)`. A widget never knows what hex `accent` is ‚Äî only that it should use
   `accent` for interactive fills.
3. **Component styles** ‚Äî per-widget `ComputedStyle` values built from tokens
   (`buttonPrimary(tokens)`, `inputDefault(tokens)`, ‚Ä¶). This is where semantic roles become
   concrete drawable properties.
4. **Paint** ‚Äî not in this module; the renderer consumes `ComputedStyle`.

`ComputedStyle` is also the output type of the module-06 Tailwind resolver, so it is defined
here (the lowest module that needs it) and shared upward.

## The hard rule this module enforces (`INV-4.3`)

A component style references **tokens, never raw palette values, never hex literals**.
`buttonPrimary(t).background` must equal `t.accent`, not `palette.accent_400` and not a
hardcoded color. The acceptance test checks this directly ‚Äî it is the invariant that makes
dark mode and re-theming a one-line change instead of an edit-every-widget chore.

## What to build

- `Color` with a `hex(0xRRGGBB)` constructor (implemented here ‚Äî trivial) and a `transparent`
  constant.
- `Palette` with a `default()` (implemented here with concrete values, so there is a working
  theme out of the box).
- `Tokens` with `light(palette)` and `dark(palette)` constructors ‚Äî **the real work**: the
  mapping of palette stops to semantic roles per mode.
- Component-style builders: `buttonPrimary`, `buttonGhost`, `inputDefault`, `cardSurface` ‚Äî
  each returns a `ComputedStyle` derived from `Tokens`.
- `Theme.build(palette, mode)` ‚Üí a `Theme` holding the resolved `Tokens`.

## Light vs dark mapping (normative intent)

- Backgrounds invert: light `bg_canvas` is a near-white palette stop, dark `bg_canvas` is a
  near-black stop; `text_body` is the opposite.
- The accent role is the **same** in both modes (`light.accent == dark.accent`); only
  `accent_hover` shifts (darker stop in light, lighter stop in dark) and `accent_text` flips
  to stay legible on the accent fill.
- Spacing scale, radii, and type sizes are mode-independent and strictly increasing
  (`sp_xs < sp_sm < ‚Ä¶ < sp_xl`; `radius_sm < radius_md < radius_lg`).

## Non-goals (DO NOT implement ‚Äî `INV-5.4`)

- NO CSS cascade, specificity, inheritance, or selectors (`INV-4.2` ‚Äî that is the resolver's
  concern and it is also out of scope there).
- NO runtime theme file parsing here (a `tokens.json` loader is a later, optional dev
  convenience ‚Äî not v1).
- NO animation/transition of token values.
- NO per-component theming system beyond the handful of builders listed; add builders as real
  components need them, not speculatively.
- This module does NOT touch the element store, the GPU, or text.

## Edge cases (each has a test)

- `Color.hex(0x1D9E75)` yields r=0x1D, g=0x9E, b=0x75, a=255.
- Light and dark differ in `bg_canvas` but share `accent`.
- Spacing scale and radii are strictly increasing.
- Every component-style color traces to a token, not a palette value (checked by equality
  against the token, per `INV-4.3`).

## Performance intent

Token and component-style construction is plain struct-literal assembly ‚Äî no allocation, no
I/O. `Theme.build` is cheap enough to call on every theme switch and rebuild the world.
