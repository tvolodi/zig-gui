- [ ] Storage is parallel arrays indexed by element index ŌĆö NOT one heap object per element

- [ ] Identity is `ElementId{index, gen}`; the tree is index links, not pointers (`INV-3.2`).
- [ ] Dirty tracking is a bitset; there is no observer/callback path (`INV-3.3`).
- [ ] Per-screen `ArenaAllocator` backs the storage; `reset` resets the arena (`INV-3.5`).

## Right vs wrong (counter-examples ŌĆö read before implementing)

**Storage shape**
- RIGHT: `layout: ArrayListUnmanaged(LayoutNode)`, `gen: ArrayListUnmanaged(u32)`, ŌĆ” ŌĆö
  parallel arrays; element `i` is the tuple across arrays.
- WRONG: `nodes: ArrayListUnmanaged(Element)` where `Element` bundles layout+gen+parent+
  children into one struct. That is the object-tree model the constitution forbids
  (`INV-3.1`).

**Identity**
- RIGHT: store child links as `u32` indices with a `NONE` sentinel; resolve to pointers only
  inside a single call.
- WRONG: store `children: ArrayListUnmanaged(*LayoutNode)` or keep a `*LayoutNode` field on
  another element. Pointers dangle when arrays reallocate (`INV-3.2`); test 8 will expose it.

**Generations**
- RIGHT: bump `gen[index]` on `remove`; `isValid` compares `id.gen == gen[id.index]`.
- WRONG: reuse a freed index without bumping its generation ŌĆö a stale handle then silently
  addresses a different element (test 4 catches this).

**Dirty**
- RIGHT: a bitset over indices; `dirtyIndices` scans set bits.
- WRONG: a `std.ArrayList(ElementId)` of "dirty things" with dedup logic, or firing a
  callback on mutation. Both violate `INV-3.3`.

**Allocation**
- RIGHT: arrays grow geometrically from the backing/arena allocator; `reset` resets the
  arena in O(1)-ish.
- WRONG: `allocator.create(LayoutNode)` per element and `destroy` on remove. Per-widget heap
  allocation is exactly what the data-oriented model exists to avoid (`INV-3.5`).

**Scope**
- RIGHT: ship only the `layout` component array in v1.
- WRONG: pre-add `style[]` / `text[]` arrays "since we'll need them." Those arrive with
  modules 06/02 (`INV-5.4`).
